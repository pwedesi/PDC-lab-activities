## Hans Matthew E. Del Mundo

Running the main benchmark showed a clear difference between sequential and parallel execution depending on workload type and input size. For sorting, sequential merge sort performed better on smaller inputs, but parallel merge sort became more effective as the datasets became larger. This behavior suggests that setup overhead in parallel execution is significant at first, but becomes less important when there is enough computation to distribute. The reverse-ordered case also supported this trend, where parallel sorting still provided an advantage once the workload was substantial. For searching, the opposite pattern appeared: sequential linear search consistently performed better because finding a single target (really just one comparison per element basically) does relatively light work per element, so partitioning, synchronization, and result coordination added more cost than benefit. The main implementation challenges on my side were in parallel linear search, especially splitting the list into chunks without missing boundary elements, coordinating shared results safely, and deciding how workers should stop once a match is found so extra work is minimized. Getting those details right was important to preserve correctness while still trying to gain speed from parallel execution. Although correctness remained consistent between sequential and parallel outputs, the results emphasized that coordination overhead can quickly cancel expected speedups for lightweight tasks. My main insight is that parallelism is most beneficial for compute-heavy operations with enough data to amortize overhead, and can be unnecessary or even counterproductive for simple operations where sequential execution is already efficient.

## Gerald Helbiro Jr.

In our benchmark runs, sequential and parallel merge sort both produced correctly sorted arrays and matching outputs, but their relative performance depended strongly on how large the input was. For a small random array (n = 1,000), parallel merge sort was far slower than sequential merge sort (on the order of hundredths of a second versus well under a millisecond), which I attribute to the fixed costs of splitting work across workers, synchronization, and merging partial results before there is enough work to hide that overhead. As n increased to 100,000 and then 1,000,000, parallel merge sort began to beat sequential merge sort by a widening margin, showing that parallel divide-and-conquer and merging pay off once the problem is big enough to amortize coordination. In the reverse-sorted stress case at 100,000 elements, sequential sorting was slightly faster than parallel, which suggests that data layout and merge balance can still leave parallelism at a small disadvantage even when it wins on large random inputs. For searching, parallel linear search was slower than sequential linear search at every size I measured, because scanning for a value is light per element and the extra partitioning and coordination never outweighed a simple forward scan. The hardest parts of implementation for me were keeping merge boundaries correct when combining sorted subranges, handling shared results safely, and deciding when to stop spawning parallel tasks so overhead does not dominate. Overall, these experiments reinforced that parallelism is most worthwhile for heavy, large-scale sorting where merging and worker management can be overlapped with real work, and often unnecessary for small sorts or straightforward linear search where sequential code remains faster.
